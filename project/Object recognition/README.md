## 데이터셋 구축 및 전처리 (Data Pipeline)

일반 객체 중심의 기존 데이터셋에는 보행 약자를 위한 핵심 클래스가 부재하여, 실제 보행 환경의 데이터를 직접 구축하였습니다.

* **클래스 구성 (총 객체 수 2,961개):**
  * 점자블록 (Braille Block): 34.3% (1,016개)
  * 사람 (Person): 23.5% (696개)
  * 자전거 (Bicycle): 13.1% (388개)
  * 오토바이 (Motorcycle): 11.3% (335개)
  * 자동차 (Car): 10.1% (298개)
  * 킥보드 (Kickboard): 7.7% (228개)

* **데이터 분할:** 총 1,510장의 이미지를 Train(70%), Validation(20%), Test(10%) 비율로 분할하여 과적합 방지
* **데이터 증강 (Data Augmentation):** 다양한 야외 조명 및 촬영 환경에 대응하기 위해 증강 기법 적용
  * 밝기 조정 (Brightness ±15%)
  * 노출 조정 (Exposure ±15%)
  * 수평 뒤집기 (Horizontal Flip)
  * 이미지 회전 (Rotation ±15°)

## 모델 학습 및 최적화 (Model Training & TFLite)

스마트폰 카메라 프레임을 받아와 딥러닝 모델이 보도 위 상황을 실시간으로 분석해야 하므로 연산 가벼움이 필수적이었습니다.

* **모델 선정:** Precision, Recall, mAP 지표와 모바일 기기에서의 평균 추론 시간(Inference Time) 및 FPS를 종합적으로 평가하여, 탐지 정확도와 처리 성능 간의 균형이 가장 뛰어난 경량 YOLO 모델 최종 채택
* **TFLite 경량화:** 안드로이드 환경의 CameraX 라이브러리와 연동하기 위해 학습된 모델을 TensorFlow Lite 포맷(`.tflite`)으로 변환
* **결과:** 서버 통신 없이 스마트폰 자체 연산만으로 실시간 객체 탐지를 구현하여 지연 시간(Latency) 최소화

## 트러블슈팅 (Troubleshooting)

**🚨 문제 상황: 정상/파손 점자블록 클래스 분리 학습 시 오분류 발생**
* **원인 분석:** 초기 설계에서는 점자블록을 `normal_braille_block`과 `damaged_braille_block` 두 가지 클래스로 분리하여 학습을 시도함. 그러나 점자블록의 색상이나 패턴은 뚜렷한 반면, 파손 부위는 시각적 차이가 매우 미세하여 모델이 전체 영역을 기준으로 분류할 때 심각한 오분류를 일으킴.
* **해결 방안:** 두 클래스를 `braille_block`이라는 단일 클래스로 통합하여 먼저 탐지 안정성을 100% 확보함. 파손 판단은 향후 `damaged_area`라는 별도의 국소 부위 탐지 모델을 앙상블(Ensemble)하는 방식으로 아키텍처를 수정하여 해결 방안 도출.
